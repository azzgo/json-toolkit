## ADDED Requirements

### Requirement: 基于结构的模拟数据生成
系统应当(SHALL)基于具有智能字段类型检测的 JSON 结构分析生成真实的模拟数据。

#### Scenario: 从 JSON 结构生成数据
- **当** 用户提供包含字段的 JSON 对象，如 `{"name": "", "email": "", "age": 0}`
- **那么** 系统生成多条记录，包含真实的姓名、邮箱地址和适当的年龄范围

#### Scenario: 智能字段名称映射
- **当** JSON 包含常见命名模式的字段 (firstName, user_email, phone_number)
- **那么** 系统基于名称模式自动将字段映射到适当的 faker.js 生成器

#### Scenario: 处理嵌套对象结构
- **当** 用户提供嵌套 JSON，如 `{"user": {"profile": {"name": "", "address": {"street": "", "city": ""}}}}`
- **那么** 系统生成真实数据，保持嵌套结构和适当的字段类型

### Requirement: 可自定义的生成参数
系统应当(SHALL)提供用户对生成参数的控制，包括数量、语言环境和字段特定选项。

#### Scenario: 可配置的生成数量
- **当** 用户指定要生成的记录数量 (1-10,000)
- **那么** 系统生成确切请求数量的模拟数据记录

#### Scenario: 特定语言环境的数据生成
- **当** 用户选择特定语言环境 (例如 'de' 表示德语，'ja' 表示日语)
- **那么** 系统为该语言环境生成文化上适当的姓名、地址和电话号码

#### Scenario: 自定义字段映射覆盖
- **当** 用户不同意自动字段类型检测
- **那么** 系统允许手动覆盖以为每个字段指定确切的 faker 方法

### Requirement: 基于 JSON Schema 的生成
系统应当(SHALL)支持 JSON Schema 输入以精确控制生成的数据约束和格式。

#### Scenario: 从 JSON Schema 生成
- **当** 用户提供包含类型定义和约束的 JSON Schema
- **那么** 系统生成完全符合模式规范的数据

#### Scenario: 处理模式约束
- **当** 模式包含最小/最大值、字符串模式或枚举值等约束
- **那么** 生成的数据遵守所有指定的约束和验证规则

#### Scenario: 模式验证
- **当** 用户提供无效或不完整的 JSON Schema
- **那么** 系统显示具体的验证错误和纠正建议

### Requirement: 高级字段类型检测
系统应当(SHALL)识别复杂字段模式并为各种领域生成适当的真实数据。

#### Scenario: 检测 ID 和引用字段
- **当** JSON 包含名为 'id'、'userId' 或类似标识符模式的字段
- **那么** 系统生成适当的唯一标识符 (UUID、连续数字等)

#### Scenario: 识别日期和时间字段
- **当** 字段以日期/时间模式命名 (createdAt, birthDate, timestamp)
- **那么** 系统在真实范围内生成适当的日期

#### Scenario: 处理类似枚举的值
- **当** 现有 JSON 包含建议枚举的重复值 (status: 'active'/'inactive')
- **那么** 系统识别模式并从检测到的值中生成随机选择

### Requirement: 数据导出和集成
系统应当(SHALL)为生成的模拟数据提供多种导出格式和集成选项。

#### Scenario: 导出为 JSON 数组
- **当** 用户生成多条记录
- **那么** 系统提供下载选项，格式为正确格式化的 JSON 数组

#### Scenario: 导出单独的 JSON 对象
- **当** 用户偏好单独对象而不是数组
- **那么** 系统提供导出为单独 JSON 对象的选项 (换行符分隔或单独文件)

#### Scenario: CSV 导出转换
- **当** 用户需要生成数据的表格格式
- **那么** 系统扁平化 JSON 结构并导出为带有适当标题的 CSV

### Requirement: 性能和可扩展性
系统应当(SHALL)高效处理大规模数据生成并提供适当的用户反馈。

#### Scenario: 大型数据集生成
- **当** 用户请求生成超过 1,000 条记录
- **那么** 系统显示进度指示器并在不阻塞 UI 的情况下生成数据

#### Scenario: 内存管理
- **当** 生成非常大的数据集 (>10,000 条记录)
- **那么** 系统使用高效的内存管理并提供流式下载选项

#### Scenario: 生成时间限制
- **当** 数据生成超过 30 秒
- **那么** 系统提供取消选项并建议减少数量或复杂性

### Requirement: 用户体验和指导
系统应当(SHALL)提供直观的界面和有用的指导以实现有效的模拟数据生成。

#### Scenario: 实时预览
- **当** 用户调整生成参数
- **那么** 系统在完全生成之前显示生成数据结构的预览

#### Scenario: 字段映射可视化
- **当** 用户想要理解或修改字段类型检测
- **那么** 系统显示 JSON 字段和选定的 faker 方法之间的清晰映射

#### Scenario: 常见模板和示例
- **当** 用户需要灵感或起点
- **那么** 系统提供常见用例的模板 (用户档案、电商产品、博客文章)

#### Scenario: 可重现的生成
- **当** 用户想要测试的一致结果
- **那么** 系统提供种子选项以在多次运行中生成相同的数据
